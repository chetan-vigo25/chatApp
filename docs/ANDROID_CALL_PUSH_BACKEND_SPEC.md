# Android Incoming-Call Push — Backend Spec (the real blocker)

**The Android client is already production-ready.** A native `FirebaseMessagingService`
(`CallMessagingService`, generated by `plugins/withCallFcmService.js`) renders the
full-screen CallStyle notification via `ExpoCallUiModule.renderIncoming()` **~100-200 ms
after the FCM arrives, before the JS runtime cold-starts** — over the lock screen, with
ringtone, vibration, DND bypass, Answer/Decline, and cold-start replay into the RN app.

So if calls don't ring instantly when the app is **backgrounded / killed / locked**, the
cause is almost always the **FCM message shape from the backend**. This spec is that
contract.

---

## The one rule that breaks everything: DATA-ONLY, HIGH priority

When the app is **backgrounded or killed**, Android routes an FCM message with a
`notification` block **straight to the system tray** and **does NOT call
`onMessageReceived`** — so `CallMessagingService.renderIncoming()` never runs and there
is **no full-screen intent, no ring**. You just get a silent tray notification.

> ✅ Send a **data-only** message (NO `notification` key) with **HIGH priority**.
> ❌ Never include a `notification` object in a call push.

## Required message

### FCM HTTP v1 (recommended)

```jsonc
POST https://fcm.googleapis.com/v1/projects/<PROJECT_ID>/messages:send
{
  "message": {
    "token": "<callee device FCM token>",
    "android": {
      "priority": "HIGH",              // wakes the device out of Doze
      "ttl": "45s",                    // a call is only useful for the ring window
      "directBootOk": true             // deliver even before first unlock after reboot
    },
    "data": {                          // ALL values must be STRINGS in v1
      "type": "call",
      "callId": "sig_<callerId>_<ms>", // your signaling id
      "callerId": "699bf91c9906da6a8930514c",
      "callerName": "User11",
      "callerImage": "https://.../avatar.webp",
      "callType": "audio",             // "audio" | "video"
      "ts": "1783072628459"            // epoch ms sent-at (stale-drop guard)
    }
    // NO "notification" key.
  }
}
```

### firebase-admin (Node)

```js
const admin = require('firebase-admin');

async function sendAndroidCallPush(fcmToken, call) {
  await admin.messaging().send({
    token: fcmToken,
    android: { priority: 'high', ttl: 45_000, directBootOk: true },
    data: {
      type: 'call',
      callId: String(call.callId),
      callerId: String(call.callerId),
      callerName: call.callerName || 'Incoming call',
      callerImage: call.callerImage || '',
      callType: call.media === 'video' ? 'video' : 'audio',
      ts: String(Date.now()),
    },
    // DO NOT set `notification`.
  });
}
```

### Legacy HTTP API (if still used)

```jsonc
{
  "to": "<token>",
  "priority": "high",
  "time_to_live": 45,
  "data": { "type": "call", "callId": "...", "callerId": "...", "callerName": "...",
            "callerImage": "...", "callType": "audio", "ts": "1783072628459" }
  // no "notification"
}
```

## Cancel / hang-up push (dismiss the ringing notification instantly)

When the caller hangs up or the ring times out, send a second data-only push so a
killed/locked callee's notification clears immediately (native
`ExpoCallUiModule.cancelIncoming`):

```jsonc
{ "message": { "token": "<token>", "android": { "priority": "HIGH", "ttl": "30s" },
  "data": { "type": "call_cancel", "callId": "sig_<callerId>_<ms>" } } }
```

## Missed call (optional, WhatsApp-style)

If the callee never answered, a `type: "call-missed"` data push (same fields + `chatId`)
surfaces the "Missed voice/video call" tray notification. The live app also de-dupes this
locally, so sending it is safe.

## When to send

- The caller hits `POST /api/v2/user/call/notify { peerId, media, callId }` — fan the
  call push to the callee's **Android devices** here.
- Your `call:ring` handler (`pushOfflineCallees`) — same.
- Per device: iOS (`pushProvider: "apns"` + `voipToken`) → **VoIP push** (see
  `IOS_VOIP_BACKEND_SPEC.md`); Android (`pushProvider: "fcm"`) → the data push above.

## Token freshness

The app (re)registers its FCM token on every boot via `setPushToken()` →
`notification:device:register`. A rebuild/reinstall rotates the token, so always push to
the **latest** stored token and prune tokens FCM reports as `UNREGISTERED` /
`registration-token-not-registered` in the send response.

## Verify (in order)

1. **Message has NO `notification` key** and `android.priority: "HIGH"`. This is the #1
   bug — a `notification`-style push can NEVER ring full-screen when killed.
2. **Send response is success** (not `UNREGISTERED`). Log FCM's response per device.
3. Test on a **physical Android device**, app **swiped away**, screen **locked** → the
   full-screen CallStyle screen must appear within ~1 s.
4. If it works with the app open but not when killed → it's almost always (1) the payload
   shape or (2) the OEM battery optimization (the app now prompts the user to disable it;
   MIUI/Vivo/Oppo also need "Autostart" enabled manually — no API for that).

## Do NOT

- ❌ Include a `notification` object in a call push (kills the full-screen path when backgrounded/killed).
- ❌ Use normal priority (Doze defers it — the call rings late or after unlock).
- ❌ Rely on the socket for killed/locked delivery — there is no socket when the app isn't running; only the FCM push can wake it.
